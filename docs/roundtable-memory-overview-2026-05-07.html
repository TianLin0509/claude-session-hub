一、为什么有这个

它解决什么

普通圆桌每次开新房，AI 都不记得你上次说过什么——你得反复讲"我喜欢结论先行""不要硬猜"等偏好。

这个系统让三家 AI 跨会话越用越懂你。

四个原则：

  • AI 自己判断什么该记（不是后台偷听全部对话）
  • 不污染答案正文（不会变成"我记得你..."复读机）
  • 记的内容是 .md 文件，你自己用 VS Code/Notepad 就能看 / 改 / 删
  • 通用圆桌的偏好和投研圆桌的偏好互不污染

设计取舍

  • AI 自主写为主，后台 worker 兜底（worker 在阶段 1）
  • 存储 = 纯 Markdown + 一个状态 JSON，0 RAG / 0 向量库，符合"半透明可见"
  • per-AI 个体记忆 + per-scene 共识（共识层在阶段 1）
  • 0 新 npm 依赖；复用 Hub 现有 hookServer + sessionManager
  • 阶段 0 仅 ~470 行新代码（store + MCP server + tests）

二、现在能做 / 不能做

能做（已验证）

  • DONE 你说"记住这个" → 三家 AI 100% 记下
  • DONE AI 在发言中觉得"该记"自己也能调工具记
  • DONE 你问"我之前说过什么" → AI 能调 list 准确说出
  • DONE 关圆桌再开（同 cwd）记忆仍在
  • DONE 三家 AI 各有自己的记忆文件，互不干扰
  • DONE 你随时能直接打开 .md 看 / 改 / 删某条

不能做 / 暂未做

  • GAP 普通业务话题下 AI 不主动 search 记忆（gate 实测 2 场都是 0/3）—— prompt 工程问题，COVENANT_GENERAL 里的 MEMORY PROTOCOL 段写的是"可以 search"软建议
  • PHASE 1 UI 角标 [已记 N] + 三个卡片按钮（个体 / inbox / profile）
  • PHASE 1 后台 checkpoint worker 每 3 轮跑 DeepSeek，派生 _profile.md 共识层
  • PHASE 1 Inbox 机制：worker 检测漏记 → 候选写到 pending-{slot}.json → AI 下次发言前自决采纳
  • PHASE 1 memory_search 命中后 recall+1 已实现，但 UI 还没显示哪条最常被引
  • PHASE 2 跨 scene 偏好（cross-scene）；阶段 0/1 锁死 scope='scene'
  • PHASE 2 90 天衰减扫描（保留删除线但不真删）

三、数据形态：文件长什么样

文件位置

每个圆桌自己的工作目录下，路径形如：

<projectCwd>/.arena/rooms/general/memory/
  ├─ pikachu.md     (slot 0 = 第一家 AI)
  ├─ charmander.md  (slot 1 = 第二家)
  └─ squirtle.md    (slot 2 = 第三家)

projectCwd 是这个圆桌的"工作目录"——通常是主驾或第一活跃 sub session 的 cwd（与 timeline.md 同源）。

每条记忆是两行：第一行元数据，第二行是内容。

路径解析

  • core/roundtable-memory/store.js: memoryFile(projectCwd, scene, slot)
  • 等价于 path.join(projectCwd, '.arena', 'rooms', scene, 'memory', slot + '.md')
  • scene 阶段 0 取值: general / research / dev（仅 general 启 memory MCP）
  • slot 取值: pikachu / charmander / squirtle（来自 core/ai-kinds.js: SLOT_IDS）
  • 隔离 Hub 模式（设了 CLAUDE_HUB_DATA_DIR）：projectCwd 默认每 meeting 独立 workspaceDir，跨 meeting 不共享 .md（除非显式传 opts.cwd）

行格式（plan §4.6）

[2026-05-07] [scope: scene] [source: explicit] [recall: 3, last: 2026-05-06] preference:conclusion-first
用户喜欢结论先行的回答模式，先给核心判断再展开论证
字段取值含义
dateYYYY-MM-DD记下时的日期（UTC）
scope scene 阶段 0/1 固定 scene。cross 留给阶段 2+
source self / inbox / explicit self = AI 在发言中自主决定记
inbox = AI 采纳了后台 worker 的候选（阶段 1）
explicit = 用户说"记住"触发
recall 整数 这条被 memory_search 命中的累计次数（list 不增加）
last 日期 / - 最后一次 search 命中的日期；- 表示从没被命中
kind:key preference|fact|observation|persisted + 短名 kind = 类型；key = 短名（同 key 写入会 dedup，更新最新内容 + recall+1）
第二行 一行内容 偏好/事实的具体描述（多行会被压成单行）

四、三个 MCP 工具

AI 看到的工具清单

每家 AI 启动时，Claude/Codex/Gemini 的 MCP 列表里多了一个 server arena-roundtable-memory，含三个 tool：

  • memory_write —— 记一条偏好/事实/观察
  • memory_search —— 搜自己之前记过的
  • memory_list —— 列出自己记过的全部

关键：三家各看自己，A 看不到 B 的 memory（per-slot 隔离）。

tool 定义入口

文件：core/roundtable-memory-mcp-server.js 的 TOOLS 数组（约 90 行的 schema）。

STUB_MODE：当 ARENA_AI_SLOT/MEETING_ID/HUB_PORT/HOOK_TOKEN 任一缺失（如用户终端独立跑 gemini，非圆桌场景），tools/list 返回空，server 仍在但不暴露任何 tool。

这避免 Gemini 全局 ~/.gemini/settings.json 注册 server 后，非圆桌场景报错。

memory_write 签名

memory_write({
  scope: 'scene',                           // 阶段 0/1 固定
  kind: 'preference' | 'fact' | 'observation' | 'persisted',
  key: 'conclusion-first',                  // 短名，同 key 写入 = 覆盖更新 + recall+1
  content: '用户喜欢结论先行，再展开论证',  // 一句话；多行会被压成单行
  source: 'self' | 'inbox' | 'explicit'     // 默认 'self'
})

memory_search 签名

memory_search({
  query: '结论',                            // 朴素子串匹配（无大小写），命中后该条 recall+1
  limit: 5                                  // 默认 5
})

memory_list 签名

memory_list({
  kind: 'preference'                        // 可选；不传 = 全部
})
// 返回的 entry 含 source / scope / recall / last 等元数据
// 关键：list 不 bump recall（区别于 search）

五、完整调用链路

从 AI 决定调工具，到 .md 文件被改写，全过程：

[1] AI 在发言中决定调 memory_write({...}) │ ▼ [2] Claude/Gemini/Codex CLI 把 tool call 转成 JSON-RPC → 写到 stdin │ ▼ [3] core/roundtable-memory-mcp-server.js（独立 child process）从 stdin 读到 │ env 含 ARENA_MEETING_ID / ARENA_HUB_PORT / ARENA_HOOK_TOKEN / ARENA_AI_SLOT ▼ [4] server 把参数 + slot + token 拼成 HTTP body → POST 到 │ http://127.0.0.1:<HOOK_PORT>/api/roundtable/memory-write ▼ [5] Hub 主进程 main.js 的 hookServer 收到 │ token === HOOK_TOKEN 校验、查 meeting、算 projectCwd ▼ [6] 调 core/roundtable-memory/store.js: appendMemoryEntry() │ 校验 kind/source/scope，dedup 同 key，写盘 ▼ [7] {slot}.md 文件被原子写入 │ ▼ [8] 主进程 sendToRenderer('memory-event') → 阶段 1 的 UI 角标会订阅这个 │ ▼ [9] HTTP 200 + JSON 结果原路返回 AI
关键设计：整条链路不经过 Electron renderer——MCP server 直接走 loopback 到 Hub 主进程的 hookServer。Hub renderer 重启 / 关 UI 都不影响 memory 写入。

六、prompt 注入：让 AI 知道有这工具

怎么"教"AI 用这个工具

三家 AI 启动时，Hub 给它们注入了一段额外的 prompt（叫 MEMORY PROTOCOL），写明：

  • 哪三类该记（preference / fact / observation）
  • 哪些不能记（单轮立场 / 临时观点 / 一次性观察）
  • 用户说"记住" → 必须立刻记，source='explicit'
  • 给了一个 memory_write({...}) 的调用示例

这段 prompt 只在通用圆桌启用（投研圆桌阶段 0 暂不接，避免和投研 MCP 冲突）。

注入位置

文件：core/roundtable-scenes.js 中的 COVENANT_GENERAL 常量末尾追加 MEMORY PROTOCOL 段（约 40 行）。

三家共用同一份 covenant（不分家定制）。

注入路径：

  • Claude family（claude / deepseek / glm / gpt / kimi / qwen）：通过 --append-system-prompt-file <promptFile> + --mcp-config <memoryMcp.json>
  • Gemini：GEMINI_SYSTEM_MD env + ~/.gemini/settings.json 全局 MCP entry + 进程级 ARENA_* env
  • Codex：codexInstructionFile + codexMcpEntries

所有三种 CLI 都拿到相同的 covenant 文本和 memory MCP server。

七、隔离 / 作用域

维度是否隔离说明
per-slot（同圆桌内三家） YES Pikachu 的 .md / Charmander 的 / Squirtle 的，三个独立文件。一家不能读另一家。
per-scene（通用 vs 投研） YES 路径含 rooms/{scene}/；通用圆桌的偏好不会跑到投研圆桌
同 scene 跨 meeting（同 cwd） SHARED 相同的 projectCwd + 相同 scene → 共享同一组 .md（关圆桌再开还在）
同 scene 跨 meeting（隔离 Hub 默认） NOT SHARED 隔离 Hub（CLAUDE_HUB_DATA_DIR 设了）默认每 meeting 独立 workspaceDir，跨 meeting 不共享。要共享需手动传 opts.cwd 给 add-meeting-sub。阶段 1 会修这个 UX。
cross-scene（通用 + 投研合并） PHASE 2+ 阶段 0/1 锁死 scope='scene'。阶段 2 才开 scope='cross' + ~/.arena/agents/{slot}/cross.md

2026-05-07 用户实测复跑：

  • 场景 1（"请记住喜欢看到分歧"）：3 家全调 memory_write，3 个 .md 创建
  • 场景 2（"Rust 重写 Python CLI 值得吗"）：0 家调任何 memory tool（普通话题不主动 search）
  • 场景 3（"我之前提过什么"）：1 家（Charmander）调 memory_list 准确引用
4 指标全过，但有一个真实 gap：普通业务话题（不带"记住"或"我之前提过什么"暗示）下，三家都不会主动调 memory_search。它们的回答风格是否符合记下的偏好，依赖 base model 性格而不是真读了 .md。这是 prompt 工程问题，COVENANT_GENERAL 里 MEMORY PROTOCOL 段写的"可以 search"是软建议。
