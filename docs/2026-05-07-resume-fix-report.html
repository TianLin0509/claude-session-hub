1. 你昨晚提的问题（根因）

"我同时打开多个 Hub 时，发现有的 Hub 中的操作，关闭后再重启 Hub，那个会话完全丢失了。"

根因（一图看懂）

时刻Hub A 内存Hub B 内存state.json 盘上说明
T0 boot[a,b,c][a,b,c][a,b,c]双 Hub 各自加载
T1 B 建 d[a,b,c][a,b,c,d][a,b,c,d]B 写盘 OK

核心问题：state.json 单文件全量覆盖 + 没有任何锁/合并机制 → last-writer-wins。Codex/Gemini 的 codexSid/geminiChatId 也只存 state.json，同样会在 race 中被覆盖为 null，导致 dormant resume 退化为 Level 2/3 fallback（context 丢失）。

关键算法 — read-merge-write

function _saveImpl(state) {
  const fd = acquireLock(LOCK_FILE);    // ← 新增锁
  if (fd == null) {
    _writeMergedToDisk(state);          // ← 锁拿不到走 fallback 不卡死
    return;
  }
  try {
    const disk = _readDiskState();      // ← 重新读盘拿别 Hub 最新写入
    const removed = _drainRemoved();    // ← 显式跟踪用户主动关闭的条目
    const merged = mergeState(disk, state, removed);   // ← LWW by updatedAt
    _writeMergedToDisk(merged);
  } finally {
    releaseLock(fd, LOCK_FILE);
  }
}

关键决策 — 自研最小化文件锁

用 fs.openSync(path, 'wx') 的 O_CREAT|O_EXCL 原子语义实现，Windows + POSIX 兼容。带 stale 探测（mtime > 10s 视为崩溃残留自动 reap）+ retry + fallback。为什么不用 npm 包： CLAUDE.md 严禁主工作目录 npm install，worktree 的 node_modules 是 junction 共享。自研版 ~110 行代码无依赖污染。

6. 关键 Bug — 由 E2E 抓到的真问题

🐛 selfHeal 把 cleanShutdown 直接翻成 false 后才返回

导致 main.js 拿到的 bootWasClean 永远是 false。renderer 收到的 wasCleanShutdown 信号永远不准——相当于每次 boot 都"上次崩溃"。

修复： 在 selfHeal 内单独保留 originalCleanShutdown，挂到返回值的 bootWasCleanShutdown 字段，main.js 读这个字段判定。

这个 bug 是单测覆盖不到的，需要真启动 Hub + 重启才暴露。E2E 价值的体现。

7. 安全考虑

  • worktree 隔离：所有改动在 C:\Users\lintian\hub-resume-fix（branch feat/resume-multi-instance-rescue），不动主仓 master。
  • node_modules 完整性：worktree 用 junction 复用主仓 node_modules，未跑过 npm install。Smoke test 启动通过（hook server listening）。

9. 不做（YAGNI）

  • 不做 vector clock / CRDT — 用户场景是加性使用，LWW 够。
  • 不做 WAL — per-id JSON 已经是 append-friendly 的备份层。
  • 不做 lockfile npm 依赖 — 110 行自研更可控、无 install 污染。
  • 不强制单实例锁 — 保留多开自由（曾经回退过单实例 commit）。
  • 不删 state.json 里的 immersiveByMeeting/pilotSlotByMeeting/dispatchModeByMeeting dict — 保留，让老 renderer 兼容。
