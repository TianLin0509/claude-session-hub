圆桌聊天模式 · 简化版决策稿

2026-05-02 · v2 · 用户洞察驱动的删减重构
"shell/卡片分离已经把架构切清楚了，私聊就是用户在子 shell 里操作，圆桌只承载协作（群策群力、主驾、副驾）。私聊模式可以直接删除，功能更鲁棒简洁。"

✓ 用户洞察的本质

  • 圆桌（meeting room）= 协作空间。所有内容公开（写 timeline + 卡片摘要）。模式只剩 dispatchMode × mode 两个正交维度。
  • 子会话区 = 私聊空间。一对一独立 PTY，圆桌完全不感知。用户想私聊就直接进对应子会话。
  • 桥接 = 用户的眼睛和复制粘贴。软件层面不做"自动归档/双向同步"——这正是之前一切复杂度的来源。

净改动估计：删除 ~600-900 行（旧 pilot mode 副产品 + 双轨私聊）+ 新增 ~150 行（dispatchMode 路由 + UI）= 整体减少 ~500 行代码。

4. 副驾评审主驾思路 · 不依赖归档的新工作流

用户最经典的诉求："让 Claude 主稿 → 让 Gemini+Codex 评审 → 切回 Claude 改稿"。在新模型下完全无需"归档"机制：

1
选定 Claude 为主驾

点圆桌 toolbar"🚗 主驾" → 选 Claude · 卡片红框出现 · dispatchMode 三态按钮解锁

2
圆桌切到"主驾发言"模式 + 普通发言（fanout）

用户在圆桌输入框发"画一个方案" · 仅 Claude 收 prompt · 内容写 Claude 卡片摘要 + timeline · 副驾两张卡片仍能在 timeline 看到 Claude 的发言（因为这是公开的）

3
（可选）用户去 Claude 子会话区跟 Claude 深聊补充细节

点 Claude 卡片"🔧 进 shell 私聊" → 进 Claude xterm → 一对一聊 · 圆桌完全不感知（这是新设计的关键 · 子会话区聊的内容只属于用户和 Claude）

4
回到圆桌切到"副驾发言"模式 + debate

用户在圆桌输入框发"@gemini @codex 你们对 Turn 2 的方案有什么意见" · Gemini+Codex 收 prompt · 副驾基于 圆桌已有 timeline（Turn 2 是 Claude 公开发言）评审 · 主驾子会话区的私聊内容副驾看不到（也不需要）

5
切回"主驾发言"让 Claude 改稿

Claude 看到副驾在 timeline 上的反馈 → 改稿 · 循环回 Step 2

关键洞察：副驾评审的对象是圆桌 timeline 上的公开发言，不是主驾的私聊。这意味着用户想让副驾评审某段思路，就在圆桌上让主驾说一遍（这是 Step 2 干的事）。私聊（Step 3）只是用户和主驾的预热/补充，不参与协作流程。

6. 你需要拍板的开放问题（v2 · 简化后只剩 4 个）

Q1 · 旧 @gemini 圆桌私聊命令彻底怎么处理？

A · 完全删除（包括快捷按钮 + 命令解析 + IPC handler）推荐
圆桌输入框不再识别 @<who>。用户想私聊就自己点 sidebar 的子会话进入。卡片"🔧 进 shell 私聊"链接保留作为快捷入口。
优点
  • 最彻底的简化
  • 语义统一（圆桌输入 = 协作 / 子会话 = 私聊，无例外）
  • 删除最多代码
缺点
  • 用户失去"@私聊"快捷键习惯
B · 改造为"导航快捷键"
用户输入 @gemini xxx → 自动选定 Gemini 为主驾 + 跳到 Gemini 子会话区 + 把 xxx 作为初始 prompt 发到 PTY。不写任何 store。
优点
  • 保留快捷键习惯
  • 仍是纯私聊（无圆桌侧记录）
缺点
  • 新增"路由 + 注入"逻辑（~50 行）
  • 语义混淆：圆桌输入框有时进协作有时跳子会话
C · 保留旧行为（kind-keyed store + 私聊 tab）
@<who> 圆桌私聊保持不变，"私聊 tab" 保留。新设计中作为"轻量单轮私聊"的入口（vs 子会话区的"长聊"）。
优点
  • 无破坏性改动
缺点
  • 圆桌依然有"半私聊"概念，违背简化原则
  • store + UI + IPC 全保留 → 代码删除量减半

Q2 · pilot mode 关闭时副作用怎么处理？（旧逻辑会归档）

A · 无副作用（仅设置 pilotSlot=null）推荐
"取消主驾"= 清除红框 + reset dispatchMode='all' + 后两个发言按钮 disabled。不调任何归档逻辑。子会话区的私聊历史就在子会话区里（用户能滚回去看），与圆桌无关。
优点
  • 逻辑最简（一行代码）
  • 用户行为可预测
  • 无意外副作用
缺点
  • 用户失去"自动 recap"工具，需要手动整理
B · 把"圆桌 timeline 的最近 N turns"打包为 recap 卡片
取消主驾时 → 取最近 N（比如 5）turns 中主驾的发言 → 用主驾自己的 LLM 生成摘要 → 写 recap 卡片到 timeline。仅打包圆桌公开内容，不涉及子会话。
优点
  • 保留"recap"价值
  • 仅基于公开内容，无私聊耦合
缺点
  • 需要保留 summary-engine 的部分逻辑
  • "打包圆桌历史"的需求是否真实存在？
C · 保留旧行为（读 sid-keyed store 生成 recap）
不删 sid-keyed store，pilot OFF 仍触发 _generatePilotRecap。和 v1 一样。
优点
  • 无破坏性改动
缺点
  • 违背简化原则（核心私聊机制还在，只是不显示）

Q3 · 切主驾时 dispatchMode 怎么处理？

A · 自动 reset 为 'all'推荐
用户切主驾（A → B 或 A → null）时，dispatchMode 强制重置为 'all'。避免"主驾切了但还在副驾发言模式，副驾对象变了"的混乱。
优点
  • 状态一致性强
  • 用户切主驾后看到熟悉的"群策群力"默认
缺点
  • 切主驾后想继续副驾发言要再点一次按钮
B · 保持当前 dispatchMode
用户切主驾时 dispatchMode 不变。如果切前是"副驾发言"，切后副驾对象自动更新，dispatch 按钮高亮不变。
优点
  • 切主驾路径更顺（不打断 dispatch 状态）
缺点
  • 用户可能没意识到副驾对象变了
  • 引入"隐式状态变化"风险
