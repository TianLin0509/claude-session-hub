① 这套系统能做什么

核心愿景一句话：每个 AI 模型在每个场景下，都成为一个会持续成长的独立伙伴。

🧠 AI 越用越懂你

每家 AI 自主写下对你的偏好/事实/理解。下次再开同场景圆桌时，AI 能自然引用之前的记忆。

🎯 场景隔离

"通用圆桌"和"投研圆桌"完全不混。投研里说"重视下行风险"，不会污染通用圆桌的代码评审。

🦋 AI 个性保留

Opus、GPT、DeepSeek 各自一份记忆，**各自独立的"老朋友"**。三家观察你的角度可能不同，这正是圆桌价值所在。

🔮 跨 meeting 延续

同 scene 关掉再开，记忆延续。这是"惊喜感"的载体——你不必再重复说偏好。

② 记忆归属关系（这是最核心的问题）

记忆按 (Scene × AI Identity) 笛卡尔积归属。下表是完整规则：

维度是否影响记忆归属解释
Scene（通用 / 投研）✅ 隔离不同领域不互相污染。投研里"看好消费"不会跑到通用圆桌
AI Kind（claude / gemini / codex / deepseek）✅ 隔离不同 AI 家族是不同伙伴
AI Model 版本（opus-4-7 / sonnet-4-6 / haiku-4-5）✅ 隔离精确到具体模型版本，每个版本是独立人格
Slot 槽位（皮卡丘 / 小火龙 / 杰尼龟）❌ 不影响仅 UI 显示，不影响记忆。换 slot 同一 AI 仍延续记忆
Meeting（具体某场圆桌）❌ 不影响同 scene 跨 meeting 共享。timeline 绑 meeting，memory 绑 scene
Project（不同项目）取决于路径scene 共享根优先（同 "general" 跨项目共享）。建议起床后实测
核心三句话
1. 每个 (Scene × AI Identity) 组合 = 一份独立记忆
2. Slot 是 UI 标签，与记忆无关——你换 slot AI 仍延续
3. 模型版本是精确到 4-7 / 4-6 / 4-5 的——切版本就换人格

③ 关键问题答疑

Opus 4.7 和 Opus 4.6 共享记忆吗？
不共享。 文件分别是 claude-opus-4-7.md 和 claude-opus-4-6.md。设计意图："每个具体模型版本是独立人格"。

影响：当 Anthropic 升级 Opus 时，新版本会"失忆"，需要重新建立。这是设计权衡，不是 bug。
未来方向（phase 4 候选）：可加入"同家族 fallback"——新版找不到时读老版。但目前精确隔离是默认行为。
Sonnet 4.6 切到 Sonnet 4.5 呢？
同上不共享。 版本号是记忆隔离的最细粒度。
Claude Opus 在通用圆桌 vs 投研圆桌共享记忆吗？
不共享。 Scene 隔离。同一个 Opus 在两个场景下被当作两个独立上下文。rooms/general/memory/claude-opus-4-7.md ≠ rooms/research/memory/claude-opus-4-7.md。
我把 Opus 从皮卡丘 slot 换到杰尼龟 slot，记忆延续吗？
延续。 Slot 仅是 UI 显示，不影响存储。Opus 在哪个槽位都读同一份 claude-opus-4-7.md。这是 phase 3 修复的核心场景。
Claude Code 单聊和圆桌共享记忆吗？
不共享，但互不冲突。 两套并行系统：
• Claude Code auto-memory：~/.claude/projects/.../memory/
• 圆桌记忆：<projectCwd>/.arena/rooms/{scene}/memory/
同一偏好可能两边都有（轻冗余可接受）。
不同项目（如 LinDangAgent vs claude-session-hub）的通用圆桌共享记忆吗？
取决于 phase 2 fallback 链。 默认 scene 共享根优先——同 scene="general" 名字跨项目共享。

这是当前最不确定的一条，建议起床后实测：在 LinDangAgent 项目开通用圆桌写偏好 → 切到 Hub 项目开通用圆桌 → 看是否共享。
如果你不希望跨项目共享，可能需要 phase 4 加 "project-scoped scene" 选项。
DeepSeek 既是圆桌成员又是后台派生器，会自己引用自己吗？
不会。 后台派生时 DeepSeek 是用 API 调用做语义聚合（输入三家 .md），不写自己的 .md。前台 DeepSeek 坐席时才自主写 deepseek-v4-pro.md。两个角色互不影响。

④ 你用起来的预期体验

第一周

  • 开通用圆桌聊真业务（让 AI 帮你看代码/写文档）
  • 偶尔点卡片右上角 📒 N 看 AI 记了什么
  • 第二天再开同 scene 圆桌——三家 AI 中至少一家会自然说出："5 月 7 号你提过 X，所以..."这就是惊喜感 ✨
  • 翻 _profile.md 看共识层——3-5 条 DeepSeek 派生的稳定偏好

第 2-4 周（信任建立期）

  • 每个 AI 模型逐渐成为"老朋友"
  • 切换 AI 模型版本（如 Opus 4.7 → Opus 4.8 升级）→ 新模型会"失忆"，需要重新建立。这是 feature 不是 bug
  • 切换 scene（通用 → 投研）→ 同 AI 重新建立场景认知

长期

  • 系统会变得"重"——每个 (Scene × Identity) 组合一份记忆，半年后可能 10-20 个文件
  • 每天后台 GC 自动清理 90 天未引用的条目（加删除线，不真删）
  • 180 天清理 inbox-archived/
  • worker 失败用 🧠 状态灯告警

⑤ 你需要重点关注的 5 个 KEY ISSUE

✅ KEY ISSUE 1：模型版本升级的"失忆体验" — Phase 4（2026-05-08）已解决
历史问题（phase 3 时存在）：Anthropic 发布 Opus 4.8 / Sonnet 4.7 时，旧版本的记忆不会自动延续到新版本。
Phase 4 修复方案：记忆从 (kind+model) 粒度回退到家族粒度——claude.md 包含 Opus / Sonnet / Haiku 全档位共享，gpt.md 包含 codex + packy-gpt 各 model 共享。同家族升级 / 切换不再失忆，跨家族（Anthropic ≠ OpenAI ≠ Google）仍隔离。
详见：docs/superpowers/plans/2026-05-07-roundtable-memory-phase4-family-report.md
⚠️ KEY ISSUE 2：跨项目共享行为不确定
phase 2 fallback 链默认 scene 共享根——同 scene="general" 跨项目共享记忆。
这意味着：你在 LinDangAgent 通用圆桌说的偏好，可能影响 Hub 通用圆桌的 AI 行为。
起床后必测：开两个不同项目的通用圆桌，验证是否串/不串，决定是否需要调整。
⚠️ KEY ISSUE 3：长期 _profile.md 漂移风险
_profile.md 上限 20 条，DeepSeek 派生质量不稳时可能"抖动"——同样的偏好这周派生进来，下周被淘汰。
监控方法：每周看一次 _profile.md。
判断标准：稳定偏好 (recall_count) 慢慢累积 = 健康；条目剧烈变化 = 派生抖动，可能需要调 prompt
⚠️ KEY ISSUE 4：AI 真自主写率仍未充分验证
当前实测："请记住"显式触发 → 100% 写入。但自然对话中 AI 主动判断该记的能力（不被告知）只有 0-1 次/场。
长期目标：系统真"越用越懂"需要 AI 自然识别——这是产品愿景的核心。
判断方法：跑 5-10 场圆桌，看 {aiKind}-{model}.md 的 source 字段——self 比例越高 = 自主性越强；explicit 占主导 = 还需要 prompt 调优
⚠️ KEY ISSUE 5：CLI 兼容性债
当前在 Claude 系列（Sonnet/Opus/Haiku）稳定验证。Gemini / Codex 圆桌接入本身有 bug 待修——多模型组合时可能出现：某家 AI 拿不到 MCP 工具 / 工具调用失败 / Hub spawn 异常等。
当前建议：圆桌优先用 Claude 系列。Gemini/Codex 接入 bug 修复后再补真实多模型 E2E。

  • 不提偏好，问纯业务问题
  • 三家 AI 至少一家自然引用昨天的偏好（"5月7号你说过..."）

⑧ 信任建立的两条线

线状态判断方法
工程证据✅ 已建立56/56 测试 PASS + 4 阶段 silent-failure-hunter + 3 路评审收敛
用户体验⏳ 需 2-3 周日常使用主观感受："AI 现在更懂我了吗？"——只能你自己回答
最关键的心法
不要把验证当任务做。phase 1+2+3 的产品愿景是"惊喜感"——它只能在你不主动看的时候出现。
每天打开 .md 找证据 → 永远找不到惊喜。
日常自然用 → 某天 AI 自然说出你忘了告诉它的偏好 → 那一刻才是真兑现。
