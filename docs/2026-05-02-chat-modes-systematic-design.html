圆桌聊天模式系统化设计

2026-05-02 · 设计探讨稿 · 决策待用户确认 · 不写代码
TL;DR — 三个核心结论
  • Shell/卡片分离后，P1 加固范围（hidden buffer / ResizeObserver / unified lifecycle）天然失效，可全部删除；P0 已 commit 的渲染修复仍保留，无需进一步加固。
  • 项目已有两套私聊 API（旧 kind-keyed `@<who>` 与新 sid-keyed 主驾私聊），新的那套从未接前端——这就是"私聊在圆桌上没有特别好的体现"的根因。
  • 建议把"主驾发言/副驾发言"（圆桌公开层）和"私聊"（子会话深聊层）拆成两个清晰的层次，两套 store 统一为 sid-keyed，归档动作（手动 + autoArchive）作为私聊→公开的唯一桥接。

2. 现状："私聊"在圆桌里的双轨制困境

2.1 项目里已有两套私聊 API

旧 API · kind-keyed

appendPrivateTurn(meetingId, kind, ...)
listPrivateTurns(meetingId, kind?)

  • 触发：圆桌里输入 @claude / @gemini / @codex
  • UI：有——圆桌侧"私聊 tab" + 快捷按钮 🤫 @私聊
  • 记录：每条只一轮（一问一答）
  • 语义：圆桌内的"轻量私聊"，副驾不感知

新 API · sid-keyed

appendPrivateTurnBySid(meetingId, sid, ...)
listPrivateTurnsBySid(meetingId, sid)
clearPrivateTurnsBySid(meetingId, sid)

  • 触发：pilot mode ON 期间主驾子会话聊天 (main.js:1380)
  • UI：无——前端从未调用过 listPrivateTurnsBySid
  • 记录：累积多轮，关闭 pilot 时 _generatePilotRecap 读出后清空
  • 语义：子会话区的"深度私聊"，归档为 recap 时副驾才看到

2.2 痛点定位

主驾私聊期间，圆桌界面只看到 placeholder「📒 主驾对话进行中」——没有轮数、没有进度、没有最新片段摘要。副驾完全感知不到主驾刚和用户私下聊了什么；要等用户关闭 pilot mode 触发 recap 后才看得见，为时已晚。

3总结 summary圆桌选定一家 PTY选定卡片 + timelineroundtable.json turns
4圆桌 @<who> 私聊圆桌选定一家 PTY仅"私聊 tab"kind-keyed store

6主驾子会话私聊主驾子会话区主驾 PTY圆桌完全不显示sid-keyed store
7主驾归档（关闭 pilot）圆桌（按钮）—recap 卡片 + segment mdtimeline + .md 文件
8独立子会话聊天（无 pilot）子会话区该 session PTY仅 shell 日志无（仅 PTY tty）
冗余识别：
  • 模式 4（@<who> 圆桌私聊）和模式 6（主驾子会话私聊）语义重叠但 store 不通——同一个用户问"@gemini 你怎么看"和在 Gemini 子会话区单独聊，记录到不同地方，互不可见。
  • 模式 5（主驾发言）会写 timeline + 卡片摘要——这是公开发言而非私聊；但 placeholder「主驾对话进行中」让人误以为 pilot ON 期间所有内容都隐藏。语义混淆。
  • 模式 7（归档）当前只在 pilot OFF 时触发，无法在不切主驾的情况下手动归档。

4. 设计方案：模式精简 + 私聊统一

4.1 三层语义分层

┌─────────────────────────────────────────────────────────────────┐ │ 层 1 · 角色层（pilotSlot） │ │ ─ 选定一家为"主驾角色"，其余为副驾 │ │ ─ 仅是标签 + UI 红框，不影响发言流 │ │ ─ 字段：meeting.pilotSlot ∈ {0,1,2,null} │ └─────────────────────────────────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────────────┐ │ 层 2 · 公开发言层（dispatchMode × mode） │ │ ─ 触发位置：圆桌界面 │ │ ─ 内容写 timeline + 卡片摘要（全员可见） │ │ ─ dispatchMode ∈ {all, pilot, observer} 决定"谁开口" │ │ ─ mode ∈ {fanout, debate, summary} 决定"怎么开口" │ └─────────────────────────────────────────────────────────────────┘ │ ▼ ┌─────────────────────────────────────────────────────────────────┐ │ 层 3 · 私聊层（sid-keyed store） │ │ ─ 触发位置：任一子会话区（不限主驾） │ │ ─ 内容写 sid-keyed store，不写 timeline / 卡片摘要 │ │ ─ 圆桌界面通过 banner / 卡片角标提示"X 正在私聊（N 轮）" │ └─────────────────────────────────────────────────────────────────┘ │ ▼ ┌────────────────┐ │ 桥接：归档 │ │ 按钮 OR auto │ └────────────────┘ │ ▼ 私聊累积 → recap 卡片 → 圆桌全员可见

5. 主驾 / 副驾 + 私聊的配合 UX

5.1 典型工作流：「深聊主驾 → 副驾评审 → 切回主驾改稿」

[Step 1] 选定主驾 = Claude ─→ 卡片红框出现，dispatchMode 解锁 [Step 2] 点 Claude 卡片"进 shell" → 进入 Claude 子会话区 ─→ 与 Claude 深聊 5 轮（写 sid-keyed store） ─→ 圆桌 banner 实时显示"🚗 Claude 私聊中（5 轮）" [Step 3] 返回圆桌（点 banner / 顶栏按钮） ─→ Claude 卡片右上角角标"私聊 5 轮" [Step 4] 圆桌按"归档片段" 按钮 (or autoArchive 自动) ─→ 触发 _generatePilotRecap ─→ recap 卡片出现在 timeline，segment md 写盘 ─→ Claude 卡片角标清零 [Step 5] 切 dispatchMode 到"副驾发言 + 辩论" ─→ Gemini + Codex 互相评审 recap 内容 ─→ 主驾卡片灰化（保留红框），副驾两张蓝框 + 脉冲 [Step 6] 切 dispatchMode 回"主驾发言" ─→ Claude 看到副驾评审 → 改稿 ─→ 进入 Step 2 下一轮深聊
