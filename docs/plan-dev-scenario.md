## 1. 背景与定位

为 Hub 圆桌新增"开发"场景，定位 **human-supervised hierarchical 协作的 advisor council**。

闭环：
clarify(并行澄清) → discuss(方案) → handoff(交接) → 单 Driver 实操 → review(后置审查) → 用户裁断 done

**关键设计原则**：
- 不做后端状态机字段，4 姿态作为 prompt 内自适应指引存在
- 不做顶部状态条/进度计数等重 UI
- 切换通过用户文字（关键词）或按钮触发，AI 永远不自动切

## 2. 核心架构

### 2.1 拓扑
| 角色 | 谁 | 职责 |
|------|-----|------|
| supervisor | 用户 | 唯一决策者，所有结论的终审 |
| advisor council | 圆桌 3 席 | 提供决策素材，禁动手 |
| worker | 单 Driver | 实操编码 |
| reviewer council | 圆桌 3 席 | 后置审查 diff |

### 2.2 软状态 4 姿态
| 姿态 | 用途 | 触发 |
|------|------|------|
| clarify | 并行 brainstorm 澄清需求 | 首轮默认 / AI 自选 |
| discuss | 方案/取舍 | AI 自选 |
| handoff | 五元组交接（两步法） | 用户文字关键词触发 |
| review | diff 后置审查 | 用户文字关键词触发 |

**轻任务直通**：边界清晰的小问题（"这个函数怎么改"），AI 可跳过 clarify 直接 discuss。

## 3. Prompt 注入分层

### 3.1 L1 永真层（每轮全量，~20 行，cache 友好放最前）

```
[开发圆桌 · 永真规则]

拓扑：用户=supervisor，你=advisor，Driver=worker。
圆桌产物是决策素材，非执行指令。

权限：允许 Read/Grep/Glob；禁 Edit/Write/workflow skill/sub-agent。

铁律：
1. 核实：涉及文件/函数/API/模块名前必须 Read 或 Grep；未核实标"(未核实推测)"。
2. 范围 (Scope Guard)：引入原始需求未提及的新模块/依赖/用户旅程标"⚠ 范围扩张"，建议收窄或拆分。
3. 收敛原则：关注信息增量而非轮数。本轮无新事实/新风险/新方案/新必答问题时，建议用户前进一步。
4. 默认假设回收：任何阶段采用的默认假设，handoff 前必须显式列出回显给用户确认；用户跳过的问题可暂按默认假设讨论，未确认前不得作为 Driver 硬前提。
5. 协作边界：可引用反驳补充他人观点（含 timeline 上轮内容），但不得视为用户已批准方案而继续推进执行。
```

### 3.2 L2a 姿态轻量层（每轮全量，~8 行）

```
[姿态自适应（自选）]
- 需求模糊 → clarify（提问为主）
- 需求明确 → discuss（路径+取舍+替代）
- 已有方案/diff → review（中性事实+风险+倾向建议）
- 边界清晰的轻任务 → 可跳过 clarify 直接 discuss
首轮默认进入 clarify。
专注你独立视角下最重要的问题，重复由用户 fan-in 减法解决。
```

### 3.3 L2b 详细规则层（按触发追注）

**clarify 详细规则**（首轮默认注入 + 用户说"brainstorm/问清楚/帮我想想"时追注）

```
[clarify 详细规则]
每位 AI 最多 5 题：
- [必答] ≤2：附 3 字段（问题 / 默认假设 / 假设错了会改变什么实现决策）
- [建议] ≤2：附 2 字段（问题 / 默认假设）
- [可选] ≤1：仅写问题
不重复 timeline 已答问题；基于已答内容追问。
用户跳过的问题视为允许默认假设讨论，handoff 前必须回显确认。
```

**handoff 详细规则**（关键词命中或用户确认模糊意图后注入）

```
[handoff 两步法]

Step 1: Decision Recall（先输出）
列出：
- 已澄清的决策点（最多 5 个最关键的）
- 用户跳过/允许默认假设的问题清单
- 当前未决问题
若 timeline 已滚动，明确声明"早期上下文可能不完整，以下回收基于现有摘要+近期轮次"。

Step 2: 五元组（用户确认 Decision Recall 后输出）
1. What — 一句话
2. Why — 业务/技术理由
3. Tradeoff — 取舍（含被否方案 + 所有默认假设及风险）
4. Open Questions — 待用户回答的决策点
   · 不能写"无"
   · 确无问题时写"不适用，原因：决策点已在 Decision Recall 列出"
5. Next Action — Driver 第一步（具体到文件/函数）
缺项不通过。Open Questions 是回流给用户裁决的，Driver 在用户回答前不开工。
```

**review 详细规则**（用户说"审一下/看 diff/复审"时注入）

```
[review 三段式]
1. 已验证事实（中性陈述，不带"好""可接受"等评价词）
   例：测试覆盖率 80% / 通过 12 个 case / 修改了 3 个文件
2. 风险/盲区/遗漏/测试缺口（必须附依据：文件/diff/日志/测试）
3. 倾向建议（可选）
   例："基于 X 已验证，倾向可接受，但 Y 仍未覆盖"
不得替用户宣布 done/merge。Driver 自审不能替代独立审查。
```

### 3.4 L3 上下文层（每轮变化）
轮次元数据 + timeline 注入（沿用现有圆桌滚动策略）+ 本轮用户问题

## 4. handoff/review 触发机制

### 4.1 关键词清单（明确触发，直接进入触发段）
- handoff: ["生成交接单", "交接", "切 Driver", "让 X 实操", "可以开工", "开始写代码", "交给 X 做", "让 X 实现", "干吧"]
- review: ["审一下", "看 diff", "复审", "帮我审", "review"]

### 4.2 模糊意图（询问确认）
AI 检测到模糊意图（"差不多了"/"整理下"等）但无关键词命中 → 输出问号确认：
> "看起来需求已经清楚了，是否生成交接单？回'是'就触发。"

### 4.3 不做自动检测
不做 diff 上下文自动切 review，不做 AI 主动猜意图后直接生成。

## 5. UI 设计

### 5.1 MVP（仅 3 项）
1. 新增"开发"场景入口（场景选择器加 tab）
2. 首次进入引导卡片（一次性，可"不再显示"）
3. 文字触发 handoff/review（无新按钮）

### 5.2 引导卡片（4 行）

```
开发圆桌：三家先帮你问清需求、讨论方案，默认只交给 1 个 Driver 实操。
你可以跳过问题；跳过项会在交接单里作为默认假设回显。
需要交接时输入"生成交接单"；Driver 改完后输入"帮我审一下"。
如需一对一深聊，可切主驾模式手动使用 superpowers brainstorm skill。

[不再显示] [我知道了]
```

- 独立"交接"按钮
- "复审"快捷按钮

## 6. Non-goals (MVP 不做)
- 不做多 Driver 并行编码
- 不做后端 devRoundtableState 持久化
- 不做自动 diff 检测切 review
- 不新增 dispatchMode
- 不复用摘要按钮作 handoff
- 不开放圆桌态 workflow skill
- 不做顶部状态条/进度计数
- 不做主驾 brainstorm 子模式

## 7. 不适用场景
以下情况建议用通用/投研场景：
- 纯讨论性问题（不涉及代码改动）→ 通用
- 投研类数据分析 → 投研
- 跟某个 AI 一对一深聊 → 主驾模式
- 审查纯文档/PPT 改动 → 通用

开发场景适用：功能实现讨论、代码方案对比、bug 分析、diff 审查、重构方案。

## 8. MVP 实施内容
1. 后端 prompt 拼装层（L1 + L2a 每轮 + L2b 按触发追注）
2. 关键词识别中间件（handoff/review 触发）
3. 场景入口 + 一次性引导卡片
4. timeline 滚动策略保持不变（沿用现有）

## 9. 验证标准
跑一周观察：
- clarify 姿态时三家按 5 题分级输出（必答 2 / 建议 2 / 可选 1）
- 用户跳过的问题在 handoff Decision Recall 中被显式列出
- review 姿态 AI 按"事实/风险/倾向"三段式输出
- 切回投研/通用场景，开发 prompt 不污染（隔离测试）
- 轻任务直通工作（"X 函数怎么改"不被强制走 clarify）
- handoff 关键词命中触发率 / 模糊意图询问确认率

## 10. 风险与回退
| 风险 | 缓解 |
|------|------|
| AI 软状态自适应不稳定 | 回退：加最小硬约束（关键词触发后强制姿态切换） |
| Decision Recall 在 timeline 滚动后不完整 | AI 必须显式声明不完整范围 |
| 用户反馈引导卡片打扰 | 可永久关闭 |
| 关键词清单漏匹配 | 后续收集真实未命中样本扩充 |

## 11. Open Questions（已确认）

- **决议**：MVP 走纯文字触发（A 方案）。1 周后看漏触发率决定要不要加按钮。

## 12. v2 → v3 变更日志
1. 删 clarify 同轮去重规则
2. 加 handoff Decision Recall 两步法（解决 timeline 滚动导致默认假设丢失）
