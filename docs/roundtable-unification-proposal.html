核心理念：一套 UI + 场景化 Prompt 注入 = 正交架构

核心理念

当前问题：UI 和场景强耦合

通用圆桌隐藏了终端（xterm），只显示卡片面板；投研圆桌同时显示终端+卡片面板。两种模式的 UI 渲染走了完全不同的代码分支，导致通用圆桌出问题时无法通过终端排查。

目标：正交解耦

圆桌只有一种 UI 形态（= 当前投研圆桌的完整 UI：卡片面板 + 终端），"通用"和"投研"只是不同的场景预设，差异仅体现在 prompt 注入（系统提示词、房间公约、MCP 工具配置）。

主要改动点

1
移除终端隐藏逻辑
删除 applyModeContainerVisibility 中 roundtableMode → mr-terminals-hidden 的分支，删除 renderTerminals 中 roundtable 的 early return。所有场景都渲染 xterm 终端。
renderer/meeting-room.js:865-894
2
合并创建入口
+ 号菜单只留一个"创建圆桌"按钮，默认以"通用"场景创建。删除 data-meeting-mode="research" 那一行。
renderer/index.html:28-29
3
简化创建流程
createMeetingByMode 不再 if/else 分支，统一走 toggle-roundtable-mode + add-meeting-sub 三家。场景切换交给 header toggle。
renderer/renderer.js:1604-1660
4
场景切换 = 纯 prompt 热替换
header toggle 点击后只重新注入 prompt/公约/MCP 配置（写新的 .md 文件 + 通知 CLI 子进程），不触发任何 UI 重建。当前的 renderTerminals 重建逻辑可以简化或删除。
renderer/meeting-room.js:124-155 + main.js IPC handlers
5
统一渲染路径
删除 renderTerminals 中 roundtable vs research 的分支判断，全部走 renderFocusMode（当前投研模式的完整渲染：卡片面板 + 三列 xterm）。
renderer/meeting-room.js:876-894

改进前后对比表

维度 Before After
+ 号菜单 🌐 通用圆桌 / 📊 投研圆桌（两项） 🎯 创建圆桌（一项）
创建后默认场景 取决于点哪个按钮 固定"通用"，进入后随时切换
通用圆桌终端 隐藏（mr-terminals-hidden） 始终显示
卡片面板 两种模式都有（一致） 保持不变
场景切换触发 重建整个终端 DOM 只替换 prompt 文件 + 通知子进程
渲染代码分支 roundtable → 卡片 / research → 卡片+终端 统一 renderFocusMode（一条路径）
新增场景成本 需同步改 UI + 创建入口 + 渲染分支 只写新的 prompt 模板 + toggle 按钮
出错排查 通用模式无终端，CLI 错误不可见 任何场景都能看到 CLI 原始输出

附加收益：场景可扩展性
